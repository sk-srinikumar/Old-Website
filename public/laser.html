Background

The lab I work in analyzes new species of extremophile bacteria. To extract one microorganism from a sample of an unimaginable amount, we utilize lasers. If the physics are right, a laser beam can freeze tiny objects in space without killing them. This is appropriately called optical tweezing or optical trapping.

This is an extremely slow process. It takes about 10 hours to capture one microorganism and move it away from an entire crowd. The laser is turned on and off multiple times throughout the process. But not really. You see, turning a laser on and off damages the laser. So you don’t mess with the power. Instead, you just use a heavy shutter to block the laser beam when you don’t have any use for it.

The optical tweezing process keeps your hands full the entire time, so being able to control the laser shutter by pressing a pedal with your foot is nice. That’s why I designed and developed the Kinesis S software.

Development

This device controls the shutter. It’s called the K-Cube. You can toggle the shutter by directly interacting with the K-Cube. But you need to be using your fingers to do that. So we thought we could hook up the K-Cube to a computer and use the manufacturer’s software to toggle the shutter using an USB pedal. The manufacturer’s software doesn’t allow using pedals with only one button. The software requires that any controller hardware has two buttons: one for opening the shutter and one for closing the shutter. We could have just bought a cheap pedal with two buttons to replace the one-buttoned pedal we already bought. But I hadn’t started my research project yet, so I had time to kill.

The biggest obstacle I faced in this project is the poor K-Cube API. The documentation was almost non existent. I used Visual Studio to analyze the API myself. The manufacturer did not code many of the declared functions. And many of the functions that were coded were not sufficient. For example, the device’s given polling function would use 99% of the CPU’s power even when the device was idle. There was no way to access attributes directly, so I had to wrap the polling function in code to add a safe delay. I spent most of my developing workarounds and wrappings.

The Kinesis S has to display a UI, give live status updates of the shutter and keep polling. To execute all these functions flawlessly at the same time, I learned asynchronous programming. You can view my clean code-job on my Github.

Design

Aiming for that sleek german Apple-esque aesthetic for all technological designs is pretty lame. Aesthetics should reflect the function of a product. We’re putting newly discovered species of extremophiles into microfluidic chips and then freezing them in 3D space with photons from a laser. That’s peak biological sci-fi baby! At the time of designing this program, I was watching a bio-sci-fi anime called Neon Genesis Evangelion. The UI of the tech in the show had a harsh 90’s aesthetic. I wanted to replicate that for my software.

The most obvious part of the UI is the schema of the devices and links utilized. They’re properly labeled and they transform to reflect the status of the devices and links. It speaks for itself. The links break and connect appropriately. The triangle indicator flashes when an input is received. I’m pretty proud of this!

Buttons are unobtrusive because they’re rarely used. Instead of bunching all the buttons together, they’re placed relative to their function. It makes sense with the schema.

There are no close, minimize and expand window buttons. This software should be visible at all times if it is in use. The window can be dragged around by holding and dragging any area that is not a button.

It’s hard to see the red filter I’ve added on everything in the UI. It gives this intense cathode tube vibe. You can see it better in the sketch file available to download on my GitHub. Check it out!
